### Architecture (Submission 1: Argus Clew)
- **Option A selected:** Scanning pipeline only. Does NOT deploy the app.
- **Hybrid approach:** Deterministic scan in CodeBuild (pattern matching script generated by the prompt) + AI reasoning layer on top of raw results
- **Why deterministic:** Reproducible scores, auditable JSON artifacts, production-ready deployment gates. Same architecture philosophy as Hermes Clew.

### Scoring Model
- **Three-dimensional:** Category weight × severity × confidence
- **Six categories (from Hermes Clew, same weights):**
  1. Semantic HTML — 25pts
  2. Form Accessibility — 20pts
  3. ARIA & Accessibility — 15pts
  4. Structured Data — 15pts
  5. Content in HTML — 15pts
  6. Link & Navigation — 10pts
- **Severity multipliers:** critical=3, major=2, minor=1
- **Confidence adjustment:** high=1.0, medium=0.7, low=0.4
- **Change metric:** Jaccard distance between consecutive finding sets
- **N/A normalization:** Categories with no applicable checks don't penalize (same as Hermes Clew scoring.py fix)
- **Score ratings:** 80-100 Agent-Ready, 60-79 Partially Ready, 40-59 Agent-Challenged, 0-39 Agent-Invisible

### AWS Infrastructure Provisioned
- CodePipeline (triggered on every commit from GitHub/CodeCommit)
- CodeBuild project (runs deterministic scan, buildspec generated by prompt)
- DynamoDB table (scan history: commit hash, timestamp, scores, findings)
- S3 bucket (HTML report dashboard, static site enabled)
- SNS topic (email notifications with one-line summary)
- SSM Parameter Store (configurable score threshold for deployment gate)
- CloudWatch dashboard (trend monitoring)
- IAM roles (least privilege, scanner role is read-only)

### Safety Guardrails (non-negotiable)
- Read-only scanning. Never modifies source code.
- Never changes dependencies.
- Never auto-fixes code.
- Never writes to the source repo.
- Never creates PRs with suggested changes.
- Awareness, not judgment. The developer fixes, the scanner verifies.

### Behavior
- **First run:** Establishes baseline. Stores scan result. Skips comparison. Notifies developer "Baseline established."
- **Subsequent runs:** Scan, score, compare to previous. Calculate delta and trends. Gate decision: deploy or pause. Send notification.
- **Email notification:** One-line summary with score, delta from last commit, top fix recommendation, link to full HTML report on S3.
- **Threshold:** Configurable via SSM Parameter Store. Developer sets their own acceptable score.
- **Rollback awareness:** DynamoDB history shows last passing score and which commit it was on.

### Levels
- **Level 1 (built):** Memory. DynamoDB history, score tracking per commit, trend charts on S3 dashboard.
- **Level 2 (built):** Pattern recognition. Surfaces patterns like "Form Accessibility drops every time files in /components/auth/ are modified." Analyzes history for recurring regressions.
- **Level 3 (described as configurable option):** Team-specific calibration. Over time, adjusts false positive detection based on codebase patterns.
- **Level 4 (future):** Predictive. Flags risky files before push based on commit history.
- **Level 5 (future):** Cross-repo benchmarking. Anonymous aggregate scores across repos.
- **Level 6 (future):** Agent simulation. Headless agent tests against staging URL.

### Audience
- Vibecoders: developers who ship fast using AI tools
- May not know ARIA, Schema.org, or why semantic HTML matters
- DO understand "an agent can't use your app because..."
- Same audience philosophy as Hermes Clew

### Report Philosophy (carries from Hermes Clew)
- Issues described from the agent's perspective, not as compliance violations
- Fixes ranked by impact with effort estimates and expected score improvement
- Confidence Notes honest about what the scan knows vs guesses
- False positive guidance per category (JSX custom components, CSS-in-JS, SPA routing, spread props)
- Stories from the agent's perspective in "What Agents Struggle With"

### Well-Architected Framework Alignment (must include)
- Security: read-only IAM, least privilege, no code modification
- Cost Optimization: on-demand DynamoDB, S3 lifecycle rules, CodeBuild only on trigger
- Operational Excellence: CloudWatch dashboards, SNS notifications, automated alerting
- Reliability: pipeline retries, error handling, DynamoDB backup
- Performance: parallel file scanning for large repos

### Troubleshooting (must include)
- IAM permission errors
- SNS email subscription not confirmed
- No HTML/JSX/TSX files found in repo
- S3 bucket name collision (globally unique)
- First run comparison edge case
- CodeBuild timeout on large repos
- DynamoDB table already exists (naming collision)
- Score threshold set too high blocking all deployments
